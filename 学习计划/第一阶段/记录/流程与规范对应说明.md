## 笔记

**2026-06-03（第1周）**

本周任务是 python3 基础语法和 js 基础语法练习。任务来源是培训清单第1周条目，目标明确：基于工具软件场景写出基础代码。开发方式是以 GSDS/ZMQ_TOOL 的实际数据为素材编写练习脚本，自测就是运行脚本看输出是否符合预期。留痕方面，代码提交到自己的git仓库的。

发现：任务确认这一步很重要——如果不先搞清楚"要学到什么程度""输出物是什么"，很容易陷入无目的的浏览文档。

---

**2026-06-10（第2周）**

本周增加了流程学习和规范学习两个任务。对照培训清单中的流程节点（任务确认→评审→开发→自测→发布/留痕），实际执行时：功能练习类任务没有走正式评审，但做了"口头确认"——明确输入文件、输出格式和处理逻辑。规范方面，"闭环"体现在每个脚本都有明确的运行验证；"留痕"体现在git commit和README记录。

---

**2026-06-20（第3周）**

本周完成了Python3数据格式处理和Vue3页面Demo。在"沟通同步"方面有体会：遇到 YAML 解析失败时，先自己排查了文件格式（发现是 `%YAML:1.0` 指令问题），定位到根因后才动代码。这个过程符合规范中的"问题升级"原则——超出能力范围时及时求助，但先要有自己的分析。

## 对应关系矩阵

| 流程/规范项 | 本阶段实际动作 | 产出物/留痕位置 | 是否符合要求 |
|-------------|---------------|----------------|-------------|
| **任务确认** | ① 从培训清单中拆出每周要做的具体条目 ② 明确每个任务的输入（真实数据文件路径）、输出（脚本/Demo/笔记）和完成标准（能运行并输出正确结果）③ 任务不清时回看清单说明和输出物模板 | `应用型软件工程师培养清单-农式曾.md`、`输出物模板.md` | 是 |
| **开发** | ① 按周创建独立目录 ② 每个脚本/组件先明确输入输出，再写代码 ③ 代码提交保持原子性——一个练习一次commit ④ 不在主干直接开发，所有改动走feature分支 | git log | 是 |
| **自测** | ① 每个 Python 脚本写完立即python3 xxx.py看输出 ② Vue3 项目：npm install → npm run dev 浏览器验证 → npm run build 确认构建无报错 ③ 数据格式脚本用本地test_data数据源验证 ④ 发现问题（如 YAML 解析失败）回修并重测 | README 中运行结果说明 | 是 |
| **留痕** | ① 所有代码提交到 git 仓库 ② 每个练习目录有 README.md 说明文件用途和运行方式 ③ 学习笔记按日期记录关键发现和踩坑 ④ 跟踪每个产出物的完成状态 | git log | 是 |
| **沟通同步** | ① 提交学习内容同步进度 ② 遇到阻塞（如 YAML 解析失败）先分析问题，定位到具体问题再大模型求助 ③ 代码完成及时push | git push 记录；每周复盘中记录的问题和大模型求助 | 是 |
