# 学习笔记：大模型辅助编程复盘

> 关联输出物模板：[输出物模板.md#3.3](../输出物模板.md)
> 目标：说明在真实工具软件项目中的可用场景、风险边界和不能直接信任的内容

---

## 笔记

**2026-06-03（第1周）**

本周在熟悉 GSDS 和 ZMQ_TOOL 代码时，用大模型辅助理解陌生模块。让它解释每个类的职责和调用链，比逐行读快很多，尤其是 ZMQ 的 PUB/SUB 模式和 multipart 消息格式，几句话就说清楚了。

注意：模型对项目内部自定义的 proto 编解码逻辑不了解——这部分需要对照协议文档。

---

**2026-06-10（第2周）**

在 ZMQ_TOOL 的日志排查场景中尝试用大模型辅助：把一段报错日志贴进去，问"这行 WARNING 可能是什么原因"。模型给出了几个方向（连接超时、序列化失败、端口冲突），虽然不能直接定位根因，但缩小了排查范围。最终定位到是simulator模块的心跳间隔配置过短。

体会：模型在日志分析中的价值是"给出可能原因列表"而非"直接给答案"，适合做第一轮粗筛。

---

**2026-06-20（第3周）**

几个实际应用场景迭代：

1. **配置迁移**：GSDS 的compile.json和device.json需要给新芯片加配置条目。让模型生成增加条目的 Python 脚本，给定现有 JSON 结构和新增参数，自动插入并保持格式一致——比自己手改少了很多遗漏。

2. **故障码映射**：ZMQ_TOOL 的fault.csv（540 条）需要按故障码和故障描述一一映射，用于页面展示故障码对应的故障描述。用大模型生成的脚本一次性出结果，比手动粘贴可靠。

## 复盘框架

| 维度 | 内容 |
|------|------|
| **可用场景** | ① **代码理解**：快速解释陌生模块的调用链和数据流（如 ZMQ PUB/SUB 模式、proto 编解码骨架）② **日志排查**：给出报错可能原因列表，缩小排查方向 ③ **配置批量处理**：对 JSON/CSV/YAML 配置做增删改查、条件筛选、格式校验 ④ **正则与格式转换**：日志字段提取、协议字段解析、跨格式数据迁移 ⑤ **接口/协议文档辅助阅读**：把技术协调卡 PDF 或 markdown 贴进去，快速画字段映射表 |
| **有效做法** | ① 先把真实数据样本（几行日志、一段 JSON、几条 CSV）贴给模型，再提需求——上下文越具体输出越准 ② 复杂任务拆成两步：先让模型解释它理解的输入结构和处理思路，确认无误再让它产出代码 ③ 对配置文件类产出，用模型生成模板+手工填入项目真实值 ④ 涉及项目内部协议/自定义格式时，先给模型看协议文档片段 ⑤ 产出的脚本和配置一律先在隔离环境跑一遍，确认无副作用再合入 |
| **风险/边界** | ① 模型不知道你的项目目录结构和文件路径——所有路径和模块名必须自己指定 ② 模型对项目内部的自定义协议、编码规则、业务逻辑完全不了解——不会比你自己读代码更准确 ③ 长上下文下模型会"遗忘"前面的约束条件——复杂任务要拆开做 |
| **使用心得** | ① 把大模型当"高级代码阅读器"和"草稿生成器"，而不是替代自己对项目的理解 ② 最适合的场景是重复性、机械性工作（批量改配置、格式转换、样板代码），不适合需要深度理解项目上下文的决策 ③ 永远先跑再信：模型产出的脚本和配置，必须在一个安全环境验证通过才能用于项目 ④ 对不熟悉的协议或格式，模型可以帮你快速建立认知框架，但精确认知必须读项目文档和源码 |
